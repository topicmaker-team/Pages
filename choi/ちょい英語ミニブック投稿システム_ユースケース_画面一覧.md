## システム前提

本システムの管理系画面(A画面群)は、すべての団体が共通で利用できる
(各団体が自団体の範囲で管理を行う)。

ちょい英語サービスの運営会社である株式会社トピックメーカーは
システム内で唯一の特別な立場にあり、同社が主催するコースは、
学習者向けの既存B画面(B-01ホーム等のコース一覧)に掲載される
対象となる。各団体がそれぞれ主催するコースを自団体メンバー向けに
一覧表示するローカルなコース一覧(学習者向け)画面は、本設計の
スコープ外である。

トピックメーカーは団体モデルの`isトピックメーカー`属性(boolean)で
識別される(true の団体はシステム内で唯一)。

### 会員と団体の関係

- 1人の会員は最大1つの団体にのみ所属する(複数団体所属は不可)
- 団体に所属しない会員(無所属会員)も存在する
- 会員は団体に所属するかどうかに関わらず、ちょい英語アカウントを
  保持してログインできる

### 会員アカウント(ちょい英語アカウント)との関係

本仕様書が扱うのは「**団体への所属関係**」の管理であり、ちょい英語
アカウントそのもの(認証情報、有料/無料ステータス等)の管理は
本仕様書のスコープ外である。

- 本仕様書の「会員削除(団体からの除外)」操作は**団体所属関係の
  解消**のみを意味し、ちょい英語アカウント自体は削除されない
- ちょい英語アカウント自体の削除は、アカウント保有者本人のみが
  実行できる(本仕様書のスコープ外)
- したがって、団体管理者が他会員のちょい英語アカウントを勝手に
  削除することはない

#### ちょい英語側の退会フローに対する制約

本仕様の団体管理モデル(団体管理者が一元的に会員管理を統制する設計)
を成立させるため、ちょい英語側の退会フローに対して以下の制約を要求する:

- **団体所属中の会員は、自身のちょい英語アカウントを自己削除できない**
  - アカウントを削除したい場合は、先に団体管理者に依頼して UC-05 で
    団体から除外してもらう必要がある
  - 除外の前段として、本人が主催コースを持つ場合は、主催者移譲
    (UC-15a)またはコース削除(UC-17)の処分が完了している必要がある
    (詳細は「データ削除の連鎖規則」参照)
- **動機**:
  - 雇用・契約関係が継続している間に本人が一方的にアカウントを消すと、
    団体管理者の管理権限と矛盾する(管理経路の迂回路を作らない)
  - 主催コースの処分を待たずにアカウントを消すと、コースデータの
    整合性が破綻する
  - UC-05(本人による自発的脱退を設けない方針)と同じ理屈で、団体所属
    の「抜け方」をすべて団体管理者経由に一元化する
- **無所属会員**(団体に所属していない会員)は、ちょい英語側の退会
  フローに従って自己削除可能

## アクター（会員のロール）

会員のロールは2つの軸で識別される。

### 軸1: 団体所属階層(投稿可否や閲覧範囲を決める)

| ロール | 説明 |
|--------|------|
| 無所属会員 | ちょい英語のアカウントを持つが、団体に所属していない会員 |
| 団体一般会員 | 団体に所属する会員。Account.団体ロール=一般 |
| 団体管理者 | 団体の登録・管理、メンバー管理を行う会員。Account.団体ロール=管理者 |
| 運営管理者 | `isトピックメーカー=true` の運営団体に所属する団体管理者。スーパーユーザー権限を持つ(下記参照) |

**運営管理者のスーパーユーザー権限**:

運営管理者は、自団体(運営団体=`isトピックメーカー=true` の団体)
の管理に加え、**他のすべての団体に対して、その団体管理者と同等の権限**
を持つ。具体的には以下の操作が可能:

- 他団体の団体情報を編集(団体名、管理部署、団体電話、外部公開ロック)
- 他団体への会員招待、会員ロール変更、会員削除
- 他団体の団体コース・団体ユニットへの介入編集・削除(UC-19, UC-28
  と同等)
- 他団体メンバーの非公開・プライベート公開・団体限定公開コンテンツ
  すべての閲覧
- 他団体の `isトピックメーカー` 属性の設定(UC-06、運用ツール経由)
- 他団体の登録・削除(団体管理者であれば自団体に対して可能な操作と同等)

運営管理者は実質的にシステム全体の最上位管理者であり、ちょい英語サー
ビスの運営者(株式会社トピックメーカー)が持つべき管理権限を表現する。

**本フェーズでの実装スコープ**:

運営管理者の他団体管理機能は、**仕様としてのみ明記**し、画面・実装は
将来フェーズで提供する。本フェーズの A 画面群は、自団体に対する管理
操作のみをサポートする。フェーズ1のドッグフーディングでは、運営団体
(株式会社トピックメーカー)の管理者は自団体に対する操作を試行すれば
十分とする。

### 軸2: コンテンツに対する関係(コースやユニットに対するふるまい)

| ロール | 説明 |
|--------|------|
| コース主催者 | コースの作成・設定・管理を行う会員。コースに対する主催関係で定義される |
| コースメンバー | コースに参加し、ユニットを投稿できる会員。コースに対するメンバー関係で定義される |
| 投稿者 | ユニットの投稿を行う会員。次のいずれか: (a) コース主催者、(b) コースメンバー、(c) 投稿者制限が「一般」に設定されたコースにユニットを投稿する任意の会員、(d) コース非帰属ユニットを投稿する任意の会員 |
| 学習者 | 投稿された教材を学習する会員 |

※ 1人の会員は軸1のロールを必ず1つ持ち、軸2のロールを文脈に応じて持つ
※ 「投稿者」「学習者」は独立した属性ではなく、会員がユニット投稿または学習を行う際の関係性として現れるロール
※ アカウント管理（登録・ログイン・プロフィール編集等）は実装済のため省略

## 公開範囲の定義

コース・ユニットの公開設定は以下の4値を取る:

| 公開レベル | 意味 |
|---|---|
| 公開 | 全ユーザーに公開 |
| プライベート公開 | URLを知っている会員のみアクセス可能 |
| 団体限定公開 | 投稿者の所属団体に所属する会員のみアクセス可能 |
| 非公開 | 投稿者本人のみアクセス可能 |

### 「団体限定公開」の対象団体

「団体限定公開」を選択した場合の対象団体は、**コース主催者または
ユニット投稿者の所属団体に自動的に固定される**(任意の団体を指定する
ことはできない)。

理由: 1会員は1団体にのみ所属するという制約と、コースは主催者の所属団体
内での共同作業の場として閉じる設計から、対象団体は一意に確定する。
他団体を指定する余地は構造的に存在しない。

UI 上の挙動:

- A-MC2 / A-MC3 / A-C2(コース): 対象団体は主催者の所属団体名を
  **参照表示**(セレクト不可、選択操作なし)
- A-MU2 / A-MU3(ユニット、コース非帰属時): 対象団体は投稿者の所属
  団体名を **参照表示**
- 投稿者・主催者が無所属の場合は「団体限定公開」の選択肢自体が
  表示されない(または disabled)。投稿可否マトリクスを参照

### 公開範囲を決める2系統: 属性と規則(制約)

公開範囲は、**コース・ユニットそれぞれが持つ「公開」属性**と、
**所属団体が課す「外部公開ロック」という規則(制約)**の2系統で決まる。
両者は外見上似た結果(コンテンツの見え方の制限)をもたらすが、
データ上の扱いが異なるため区別する。

| | 属性 | 規則(制約) |
|---|---|---|
| 例 | `コース.公開`、`Unit.公開` | `団体.外部公開ロック` |
| データの持ち方 | 実体が値を保持する | 設定値は持つが、適用結果の値はDBに格納しない |
| 値の確定タイミング | 変更操作時にDBに書き込み | 閲覧・配信時に動的評価 |
| 名称規則 | 「公開ステータス / 公開設定」 | 「ロック」 |

具体的な現れ方:

- **コース帰属切替時の公開値の上書き**(後述) → **属性更新**。
  Unit.公開 のDB値を実体的に書き換える(投稿者の意思を反映する変更操作)
- **団体の外部公開ロックによる格下げ**(後述) → **規則(制約)の適用**。
  Unit.公開 や コース.公開 のDB値は書き換えない。閲覧・配信判断時に
  動的評価される

> 規則(制約)変更時の既存コンテンツの扱い(例: `off` → `全員ロック`に
> 切り替えた時点で既に「公開」で投稿されている既存ユニット)は、
> 配信時の動的評価で自動的に格下げされる。投稿者の指定値(DB上の
> `Unit.公開`)は書き換えない。

**コース帰属ユニットの公開範囲**:
コース帰属ユニットの実効的な公開範囲は、所属コースの公開設定に従う。ユニット自身の公開設定は「公開 / 非公開」のみをUIで選択可能とし、コースの公開範囲内で当該ユニットを公開するか否かを決定する。

**コース非帰属ユニットの公開範囲**:
ユニット自身が4値すべてを選択可能。

### ユニットの公開属性のデータ保持

ユニットの`公開`属性はモデル上は常に4値enum(公開 / プライベート公開 / 団体限定公開 / 非公開)として保持する。UIでの選択可能な値はコース帰属状態に応じて2値または4値に絞られるが、DB層では常に4値のいずれかが格納される。

### コース帰属状態の変更時の公開値の扱い(属性更新)

ユニットのコース帰属状態が切り替わる際の`公開`値は、以下のルールに従う。これは**属性更新**であり、Unit.公開 のDB値が実体的に書き換えられる。

**コース非帰属 → コース帰属 への切替時**:

- ユニットの公開値が「非公開」の場合: コース側の公開値にかかわらず「非公開」のまま不変
- ユニットの公開値が「公開 / プライベート公開 / 団体限定公開」のいずれかの場合:
  - コース側の公開値と一致していれば不変
  - 不一致であればコース側の公開値でユニットの公開値を強制上書きする

具体例:

| 切替前のユニット公開値 | コースの公開値 | 切替後のユニット公開値 |
|---|---|---|
| 非公開 | (任意) | 非公開(不変) |
| 公開 | 公開 | 公開(不変) |
| 公開 | プライベート公開 | プライベート公開(上書き) |
| 公開 | 団体限定公開 | 団体限定公開(上書き) |
| プライベート公開 | 団体限定公開 | 団体限定公開(上書き) |
| 団体限定公開 | 公開 | 公開(上書き) |

上書きが発生する場合、UI側では保存前にユーザーに確認を行うことが望ましい(警告モーダル等)。

**コース帰属 → コース非帰属 への切替時**:

- DBに保持されている公開値をそのまま引き継ぐ
- UIは4値ラジオに切り替わり、現在のユニット公開値を初期選択として表示する

**新規投稿時(A-MU3)**:

- 切替前の値が存在しないため、コース帰属を選択した時点でUIは2値(公開/非公開)のみを表示し、ユーザーが選択した値を保存する

### 団体の外部公開ロック(規則(制約))

団体管理者が設定する団体ポリシー `団体.外部公開ロック : enum` (デフォルト `off`) により、その団体所属メンバーが投稿するコンテンツの実効的な公開範囲を制限できる。これは**規則(制約)** であり、各投稿の Unit.公開 や コース.公開 のDB値は書き換えない。閲覧・配信時に動的に評価される。

**3値enum**:

| 設定値 | ロック対象 | 動機 |
|---|---|---|
| `off` | 誰もロックされない | デフォルト。秘匿要件のない団体 |
| `全員ロック` | 一般会員 + 団体管理者 | 秘匿性の高い教材を扱う団体。組織として外部公開を一律制限 |
| `一般会員のみロック` | 一般会員のみ(管理者除外) | 一般会員の誤投稿を防ぎつつ、団体管理者は広報的な教材を外部公開できる運用 |

**動機**:

- 秘匿性の高い教材を団体外に出さないようにする(秘密保持)
- 加えて、団体内向け教材とは別に**外部向けの広報的な教材**を団体管理者が直接作成・外部公開する運用に対応(`一般会員のみロック`)

**効果**: 一方向(送信側のロック)。**ロック対象の投稿者**のみ実効公開範囲が変化する。閲覧側は影響を受けない。

ロック対象の投稿者の挙動:

| 投稿者の指定値 | ロック非対象時 | ロック対象時 |
|---|---|---|
| 公開 | 公開(変化なし) | **団体限定公開**(格下げ) |
| プライベート公開 | プライベート公開(変化なし) | **団体限定公開**(格下げ) |
| 団体限定公開 | 団体限定公開(変化なし) | 団体限定公開(変化なし) |
| 非公開 | 非公開(変化なし) | 非公開(変化なし) |

**実効公開状態の計算**:

```
ロック対象判定:
  外部公開ロック = off                  → 誰もロック対象でない
  外部公開ロック = 全員ロック           → 全メンバー(管理者含む)がロック対象
  外部公開ロック = 一般会員のみロック   → Account.団体ロール=一般 のみロック対象

実効公開状態:
  投稿者がロック対象でない → 投稿者の指定値そのまま
  投稿者がロック対象である → min(指定値, 団体限定公開)
    つまり「公開」「プライベート公開」は「団体限定公開」に格下げ、
    「団体限定公開」「非公開」は変化なし
```

**UI 上の挙動**(投稿フォーム A-MU2/A-MU3, A-MC2/A-MC3):

- ロック対象の投稿者が「公開」「プライベート公開」を選択した時点で、警告メッセージを表示する。例:

  > あなたの所属団体「XX」は外部公開ロックが有効です(全員ロック)。
  > この投稿は実効的に「団体限定公開」として扱われ、団体外には公開されません。

  または `一般会員のみロック` モードで一般会員が投稿する場合:

  > あなたの所属団体「XX」は外部公開ロックが有効です(一般会員のみ)。
  > 一般会員の投稿は実効的に「団体限定公開」として扱われます。
  > 団体外への公開が必要な場合は団体管理者にご相談ください。

- ロック対象でない投稿者(`一般会員のみロック` モードの管理者など)には警告を表示しない
- 公開状態のラジオボタンの「公開」「プライベート公開」を選択不可にする選択もあり得るが、**選択は許可した上で警告**する方が実装上シンプル(ロック設定が後で変わったときに自動的に正しい範囲に拡大される)

**対象範囲**: コース、ユニット(コース非帰属ユニットの場合)。コース帰属ユニットは所属コースの公開設定に従うため、コース側でロックが効いていれば自動的にユニットも影響を受ける。

**閲覧側の挙動**:

ロックの種類にかかわらず、他者の閲覧挙動は影響を受けない(投稿の実効公開状態が「団体限定公開」相当に格下げされている結果として、団体外からは閲覧不可になる)。すなわち閲覧権限ロジックは実効公開状態に基づいて従来通り評価される。

### 投稿可否マトリクス(まとめ)

ユーザーロールごとに、各公開状態での投稿可否を整理する。

| 公開状態 | 無所属会員 | 団体一般会員 | 団体管理者 | 運営管理者 |
|---|---|---|---|---|
| 公開で投稿 | ○ | ○(*1) | ○(*2) | ○ |
| プライベート公開で投稿 | ○ | ○(*1) | ○(*2) | ○ |
| 団体限定公開で投稿 | ×(団体所属なし) | ○ | ○ | ○ |
| 非公開で投稿 | ○ | ○ | ○ | ○ |

(*1) 所属団体の外部公開ロックが `全員ロック` または `一般会員のみロック` の場合、実効的に「団体限定公開」相当に格下げされる。投稿者には警告が表示されるが、設定操作自体は可能。

(*2) 所属団体の外部公開ロックが `全員ロック` の場合、実効的に「団体限定公開」相当に格下げされる。`一般会員のみロック` の場合は管理者は対象外で格下げなし(広報的な教材を団体管理者が外部公開する用途を想定)。

### 閲覧可否ロジック

投稿の**実効公開状態**(投稿者指定値に団体ロックを適用した結果)に基づいて以下のように評価する。

| 実効公開状態 | 閲覧可能なユーザー |
|---|---|
| 公開 | 全ユーザー |
| プライベート公開 | コースメンバーのみ |
| 団体限定公開 | コース対象団体のメンバー + コースメンバー |
| 非公開 | 投稿者本人のみ |

**例外規則**:

- 団体管理者は、自団体メンバーの投稿について、介入編集権限により閲覧可能(UC-19, UC-28)
- 運営管理者(`isトピックメーカー=true` 団体の管理者)は、全投稿閲覧可能(他団体メンバーの非公開・プライベート公開・団体限定公開コンテンツも含めて閲覧・編集可能。詳細はアクター定義の「運営管理者のスーパーユーザー権限」参照)

## 教材階層のIDの二系統(公開ID と 編集ID)

教材階層(パート/デッキ/カード/センテンス)の各要素は、目的の異なる
2系統のIDを持つ。実装時のID設計と、投稿データ(エクセル)更新フロー
の理解のために、両者を区別する。

### 公開ID (Public ID)

- **目的**: URL用、外部公開される識別子
- **保持する階層**: パート、デッキ
- **用途**: 学習者向けB画面でURLパスとして使用される
  (例: `/decks/{デッキ公開ID}` のような形)
- カード、センテンスは公開IDを持たない(外部からの個別参照を想定しないため)

### 編集ID (Editing ID)

- **目的**: 投稿データ(エクセル)の行同定用、内部のみで使用
- **保持する階層**: パート、デッキ、カード、センテンス(全階層)
- **用途**: 投稿データ更新(UC-22)時の create / update 判定に用いる
  - エクセルにダウンロード時、各行に編集IDが付与される
  - アップロード時、編集IDが空欄の行は新規作成、入っている行は更新と判定
- 詳細(エクセル列構成、記入ルール)は `投稿データ_メディアパック仕様.md`
  を参照

### 投稿者にとっての扱い

- 初回投稿時はすべて空欄でよい(システムが投稿成功時に発行)
- 更新時は、UC-23 でダウンロードしたエクセルを編集してアップロードする
- 投稿者向けガイド: 「**編集ID列は触らないでください**」と
  ヘルプテキストおよびエクセル列ヘッダのコメントで案内

## ユースケース一覧

### 1. 団体管理（団体管理者）
- UC-01: 団体を登録する
  - 前提: ちょい英語のアカウントでログイン済みの会員が実行
  - 前提: 実行会員は団体未所属であること(1会員は1団体のみ所属)
  - 事後条件: 登録された団体が作成され、実行会員がその団体の
    デフォルト管理者となる(Account.団体ロール=管理者)
  - 管理者権限は後でUC-04により変更可能(招待した他の会員を
    管理者にすることもできる)
- UC-02: 団体情報を編集する（団体名、管理部署、電話、外部公開ロック）
  - 外部公開ロックは 3値 enum 属性 (`off` / `全員ロック` /
    `一般会員のみロック`) で、`off` 以外にすると当該団体メンバーの
    投稿の実効公開状態が「団体限定公開」を上限に制限される
  - 詳細は「公開範囲の定義」セクション「団体の外部公開ロック」を参照
- UC-03: 会員を招待・追加する
  - 招待対象は団体未所属の会員のみ(1会員は1団体のみ所属の
    制約による)
- UC-04: 会員のロールを変更する（管理者 / 一般）
  - **最後の管理者制約**: 自分自身を一般に降格しようとして、結果として
    団体管理者が0人になる場合はエラー。先に他会員を管理者に
    昇格させる必要がある(下記 UC-05 と同じ制約)
- UC-05: 会員を団体から削除する(団体からの追い出し)
  - **本UCの意味**: 団体所属関係の強制解消。Account.所属団体 を null
    にする操作であり、ちょい英語アカウント自体は削除されない
    (詳細は「会員アカウント(ちょい英語アカウント)との関係」参照)
  - 主アクター: 団体管理者
  - 効果:
    - 対象会員の Account.所属団体 を null にする(無所属化)
    - 対象会員が他のコースのコースメンバーであった場合、その関係も解消
      (主催者と同じ団体にいなくなるため)
    - **対象会員が投稿したユニットの所有権を、種別に応じて処理する**
      (下記「投稿ユニットの所有権処理」参照)
  - **制約**:
    - 対象会員が団体管理者で、かつ削除すると管理者0人になる場合はエラー
      (先に他会員を管理者に昇格、または UC-04 でロール変更)
    - 対象会員が主催コースを持つ場合は、**事前に主催者移譲(UC-15a)
      またはコース削除(UC-17)が完了している必要がある**
      (団体管理者は介入編集権限の延長として代理で移譲・削除操作を
      行うことができる)。この制約により、団体未所属になった会員が
      コースを保持し続けることはない
  - **投稿ユニットの所有権処理**:
    対象会員が投稿したユニットは、種別および団体の外部公開ロック設定に
    応じて以下のように処理する。動機: コースは団体内の共有物であること、
    および外部公開ロック適用下では団体所属中に作成したコンテンツの
    持ち出しを禁止すること。

    | 条件 | コース帰属ユニット | コース非帰属ユニット |
    |---|---|---|
    | 外部公開ロック ≠ off | 当該コースの**主催者**へ所有権移譲 | **団体管理者(指定)**へ所有権移譲 |
    | 外部公開ロック = off | 当該コースの**主催者**へ所有権移譲 | 除外者のアカウントに**残る** |

    - 「コース帰属ユニットの主催者」は、除外実行時点での当該コースの
      主催者を指す(除外対象が主催者のコースは事前条件で排除済みなので、
      除外対象 ≠ 主催者)
    - 外部公開ロック ≠ off の場合の「団体管理者(指定)」は、UC-05 実行
      時に指定する。団体管理者が1人しかいない場合はその人にデフォルト
      で決まる。複数いる場合は UI で指定させる
    - 所有権移譲の効果: Account → Unit [投稿] の関連を付け替え、新所有者
      の「マイユニット」一覧に表示される。所有権移譲されたユニットの
      コース帰属関係(Unit が属する コース)は維持される
    - 公開設定の値は所有権移譲によって変化しない(規則による格下げの
      評価は新所有者の所属団体・ロールで再計算される)
  - 補足: 対象会員のユニットのデータについて、上記表で「除外者の
    アカウントに残る」となるもの以外は所有権が移譲されるため、除外者の
    マイユニットには含まれなくなる。詳細は本セクション末尾「データ削除
    の連鎖規則」参照

> **本仕様には「本人による自発的な団体脱退」のユースケースは設けない。**
> 団体への加入(UC-03)・ロール変更(UC-04)・除外(UC-05)はすべて団体
> 管理者が主導する設計であり、一般会員が一方的に団体から抜ける操作は
> この管理モデルと整合しない(実運用でも、雇用・契約関係の解消は組織側
> の手続きが起点になる)。会員側で団体を抜けたい意図がある場合は、
> 団体管理者に依頼して UC-05 を実行してもらう。

### 3. ユニット投稿（投稿者）

ユニット投稿には、エクセル(教材データ)・音声パック(zip)・画像パック(zip)
の3種のファイルが関わる。エクセルは14列構成(編集ID 4列 + 表示用10列)
で、詳細は `投稿データ_メディアパック仕様.md` を参照。

**投稿フローの基本方針**: 「**本登録 → 自動生成 → A-MU2 でレビュー(必要なら修正)**」

A-MU3 では教材データとユニット定義を入力し、「投稿する」(または
「下書き保存」)で本登録する。本登録の直後、サーバー側で TTS / AI画像
の自動生成が非同期で開始される。完成したコンテンツは A-MU2 上で
レビューでき、必要なら修正できる。

このフローの主眼は、**サーバー側生成処理を伴うコンテンツでは登録前
プレビューが原理的に成立しないこと**である(生成物がまだ存在しない
ため)。したがって A-MU3 から登録前のプレビュー画面は廃止し、
本登録後のコンテンツ確認は A-MU2 が担う。

公開設定は A-MU3 / A-MU2 のいずれでも設定・変更でき、A-MU3 投稿時の
「下書き保存」は公開設定を「非公開」固定にする利便機能として
従来通り残る。

詳細は UC-20 / UC-21 を参照。

- UC-20: ユニットを新規作成・投稿する
    - ユニット定義（ユニット名、紹介、カテゴリー、サブカテゴリー、フリータグ、画像、公開設定、料金設定）
    - エクセルファイルのアップロード(14列構成、`投稿データ_メディアパック仕様.md` 参照)
    - **メディア供給モードの選択**(音声・画像それぞれ独立に、排他選択):
      - メディアパックモード: 該当 zip をアップロード
      - 自動生成モード: TTS / AI画像で生成(zipアップロードなし)
    - エクセルとzipの不整合(モード不一致、参照されているファイルがzipに
      ない、余剰ファイル等)はエラーとして投稿を拒否する
    - **登録前プレビュー画面は提供しない**(生成処理が完了するまで
      コンテンツ全体は確定しないため)
    - 投稿時の挙動:
      1. 投稿者が「投稿する」(または「下書き保存」)ボタンを押下
      2. システムがバリデーション(エクセル/zip整合性、モード一貫性、
         スキーマ制約)を実行
      3. ユニットIDを発行、DBに保存
         - 「投稿する」: 投稿者が選択した公開設定で本登録
         - 「下書き保存」: 公開設定を「非公開」固定で本登録
      4. メディア供給モードに応じた処理:
         - メディアパックモード: zip を S3 `packs/` に保存 + 展開
         - 自動生成モード: TTS / AI画像生成ジョブを非同期投入
      5. A-MU2 編集画面に遷移
    - 自動生成の完了は A-MU2 上で表示される(進捗表示、完了通知)
- UC-21: ユニット情報を編集する
    - ユニット定義(名称、紹介、カテゴリ等)、コース帰属、公開設定、
      料金等を編集する
    - 自動生成された音声・画像のレビュー、必要に応じた差し替え(UC-22)
      もここで行う
    - 公開設定の変更も通常の編集の一環として可能
      (例: 投稿時「非公開」で本登録した下書きを、レビュー後に「公開」
      に切り替える、など)
    - コース非帰属時は4値、コース帰属時は2値(公開/非公開)を選択可能
- UC-22: 投稿データを更新する（差分処理）
    - 投稿データ(エクセルファイル)内の行データの一部または全部を編集する
    - エクセルファイル自体のバージョン管理は行わない
    - **メディアパック(音声zip / 画像zip)は一括差し替え方式**
      (個別ファイル単位の差分処理は行わない)
    - **メディア供給モードの切替も可能**: A-MU2 上で音声・画像のモードを
      自動生成 ↔ メディアパック の間で切り替えられる。切替時は既存
      メディアを全削除して新モードに置き換える(投稿データ・メディア
      パック仕様書 §3.5 参照)
    - 差分処理(create/update/ignore の判定、編集IDによる行同定、S3操作、
      モード切替時の挙動)の詳細は `投稿データ_メディアパック仕様.md` 参照
- UC-23: 投稿データをダウンロードする
    - DLされるエクセルには現在の編集ID(パート/デッキ/カード/センテンス)
      および現在のメディアファイル名(モードに応じてユーザー命名または
      システム生成名)が反映されている
    - メディアの一括ダウンロードもこのUCに含む: **モードを問わず S3
      `packs/` 配下の zip を取得**(自動生成モードでも、ジョブ完了時に
      zip が作成されている)
    - 自動生成モードでジョブ未完了の場合はダウンロード不可
    - これにより、自動生成で本登録 → ダウンロード → 手元編集 →
      メディアパックモードへ切替アップロード、という運用フローが可能
- UC-24: ユニットをコースに紐づける
    - 1つのユニットは高々1つのコースにのみ紐づく

## データ削除の連鎖規則

各エンティティが削除された場合の連鎖の挙動を整理する。本フェーズでは
本仕様書のスコープ内(団体所属関係およびそれに紐づくコース・ユニット
等のデータ)のみを扱い、ちょい英語アカウント自体の削除は対象外。

### 団体が削除された場合

団体削除UCは本フェーズでは UI 提供せず、運用操作扱いとする(運営管理者
の UC-65 として将来実装予定)。挙動の仕様は以下:

- **会員** → 無所属化(Account.所属団体 を null にする。アカウント自体は残る)
- **団体所有コース**(団体所属メンバーが主催するコース) → 削除
  - 削除されるコースの**コース帰属ユニット** → 連鎖削除
  - 連鎖削除されるユニットのメディアファイル → S3 から削除
- **コース非帰属の個人ユニット** → 削除されない(会員のアカウントに残る)
  - ただし、**公開設定が「団体限定公開」のユニットは「非公開」に
    切り替える**(参照先団体が消滅するため、団体限定公開という公開設定
    が意味を失う。安全側に倒して非公開にする)
  - 「公開」「プライベート公開」「非公開」のユニットは公開設定を維持

事前にユーザーは UC-23 等で投稿データを退避できる(退避はユーザー責任)。
削除前の確認モーダルで配下件数を強く警告する。

### コースが削除された場合(UC-17 / UC-19)

- **コース帰属ユニット** → 連鎖削除
- 連鎖削除されるユニットのメディアファイル → S3 から削除
- **コースメンバー関係** → 削除(関係レコードのみ。会員アカウントは残る)
- **お気に入り・コメント**(コース対象、コース帰属ユニット対象、その配下
  のパート対象、デッキ対象) → 削除

事前にユーザー(主催者・メンバー)は UC-23 等で投稿データを退避できる
(退避はユーザー責任)。削除前の確認モーダルで強く警告する。

### 団体管理者(会員)が削除される場合(UC-04 / UC-05)

- **削除/ロール変更**いずれも、最後の管理者(自分が抜けると管理者
  0人になる)はエラー
- 最後の管理者は他の会員を管理者に昇格させてから操作する必要
- 一般会員が0人の団体で唯一の管理者を抜きたい場合は、団体削除を選ぶ
  (上記「団体が削除された場合」へ)

### コース主催者(会員)が団体から削除される場合(UC-04 はロール変更のため対象外、UC-05)

- 削除前に、主催コースについて主催者を別の人(同団体内の他会員)に
  **移譲(UC-15a)するか、コースを削除(UC-17)するか** のいずれかを
  完了している必要がある
- 完了するまで削除操作はエラー
- 団体管理者が代理で移譲または削除の操作を行う(介入編集権限の延長として)
- この制約により、団体未所属になった会員がコースを保持し続ける状態は
  発生しない

### 除外される会員(コース主催者でない場合も含む)の投稿ユニットの所有権処理(UC-05)

団体から除外される会員が投稿したユニットの所有権は、種別および団体の
外部公開ロック設定に応じて以下のように処理する。詳細は UC-05 の
「投稿ユニットの所有権処理」を参照。

| 条件 | コース帰属ユニット | コース非帰属ユニット |
|---|---|---|
| 外部公開ロック ≠ off | 当該コースの主催者へ所有権移譲 | 団体管理者(指定)へ所有権移譲 |
| 外部公開ロック = off | 当該コースの主催者へ所有権移譲 | 除外者のアカウントに残る |

動機: コースは団体内の共有物として団体側で守る。外部公開ロック適用下
では団体所属中に作成したコンテンツの持ち出しを禁止する。

### 会員(ちょい英語アカウント自体)が削除された場合

ちょい英語アカウント自体の削除は本仕様書のスコープ外であり、
ちょい英語側の本人による退会フローで実施される。

本仕様書としては、ちょい英語側に対して以下を要求する(詳細は「会員
アカウント(ちょい英語アカウント)との関係」参照):

- **団体所属中の会員はアカウントを自己削除できない**ように、ちょい英語
  側の退会フローでブロックする
- アカウント削除には、先に UC-05 による団体からの除外が完了している
  必要がある(さらにその前段として、主催コースの処分: 移譲 UC-15a
  または コース削除 UC-17 が完了している必要がある)
- 結果として、ちょい英語アカウントが削除されるタイミングでは、当該会員
  は団体未所属の状態であり、コース主催関係も持たない。本仕様書の管理
  データに対する追加の連鎖処理は発生しない
- 当該会員が投稿していたユニット(個人保有のマイユニット)の扱いは、
  ちょい英語側の退会フローで定義する(本仕様書のスコープ外)

## S3 データの公開範囲(フェーズ別方針)

メディアファイル(音声、画像)の S3 公開範囲は、フェーズに応じて
段階的に厳格化する。

### フェーズ1: トピックメーカー社内ドッグフーディング

- S3 バケットは **全公開** とする
- メディアURLは固定の S3 公開URL でクライアントから直接参照可能
- 社内ドッグフーディング期間中は外部公開ロック等の規則(制約)による
  「閲覧不可化」が完全には機能しない(URL を知っていれば誰でも見える)。
  ただし社内利用のみのためリスクは限定的

### フェーズ2: 他団体受け入れ可能化以降

- **署名付きURL方式** に移行する
- メディア配信時に動的に署名付きURLを発行し、有効期限と
  閲覧権限のチェックを伴う形で配信する
- 外部公開ロック(規則(制約))・コース公開設定・ユニット公開設定(属性)が、
  メディアファイルの実際の閲覧可否にも有効に効くようになる
- メディアURLの形式が変わるため、**クライアント側でメディアURLを
  ハードコードしない設計を推奨**

### 設計上の含意

外部公開ロック機能の真の効果はフェーズ2の署名付きURL導入で発揮される。
フェーズ1では「投稿者の意思を妨げない範囲で運用上のガードレールを
設ける」程度の効果に留まることを、運用関係者は了解しておく必要がある。

## 補足事項

- **パートの扱い**: 既存「ちょい英語」のデータ構造との互換性のため、ユニット内部にパート層を持つが、「ちょい英語ミニブック」では1ユニット=1パート固定とし、UIには露出しない
- **料金設定**: 課金システムは既存「ちょい英語」を利用。ユニット側は`無料 / 有料`フラグのみ保持
- **投稿データのバージョン管理**: エクセルファイル自体のバージョン管理は行わない（常に最新版のみ保持）。メディアパック(音声zip / 画像zip)も同様にバージョン管理せず、最新版のみ保持(一括差し替え方式)
- **コース・ユニットの所有関係**: コース・ユニットは会員個人が主催/投稿する(団体の所有物ではない)。団体所属の有無に関わらず、会員はコース・ユニットを作成可能。団体管理者は自団体メンバーのコース/ユニットに対して集約管理ビュー(団体〇〇)からアクセスでき、編集・削除などの上位権限を持つ(UC-18, 19, 27, 28)
- **介入編集の履歴**: 団体管理者が他メンバーのコース/ユニットに介入編集を行った際の履歴ログは、本フェーズ対象外(将来拡張点)
- **フェーズ1運用**: 本設計はトピックメーカー社内ドッグフーディング運用を前提とする。社員が各種ロール(無所属/団体一般/団体管理者)のアカウントを使い分けて、フェーズ2で想定される全ユーザー体験を事前検証する。アクセス入口はトピックメーカー専用暫定プロフィール画面(A-P1、URL直接入力)
- **S3 アップロード方式**: クライアントから直接 S3 にアップロードするのではなく、**アプリサーバー経由(プロキシ方式)** で S3 に転送する。バリデーションをサーバー側で確実に実行するため、および AWS 認証情報がクライアントに漏れないため。詳細は `投稿データ_メディアパック仕様.md` 参照

### 招待を受けた側のユースケース/画面(設計済み)

**経緯**:
従来 UC-03「会員を招待・追加する」は団体管理者側の操作のみが
記述されていたが、被招待者側のフローが抜けていた。
これを別途フローチャートとして設計し、本仕様書とは別ドキュメントに
まとめた。

**成果物**:
- `招待フロー_団体.md`(団体招待: UC-03/50/51/52)
- `招待フロー_コース.md`(コース招待: UC-15 + 被招待者側)

両ドキュメントには Mermaid フローチャート、状態遷移図、状態別の
A-M1/A-CM1 表示仕様、および設計判断が必要な事項の一覧を含む。

**正式ユースケース番号**:
- UC-50: 団体への招待メールを受け取って承諾する(アカウント既存)
- UC-51: 団体への招待メールを受け取ってアカウント作成と承諾を行う
- UC-52: 招待状態を管理する(団体管理者側、A-M1での表示)
- UC-15 拡張: コースメンバー招待の被招待者側フロー
  (アカウント既存・未取得・URL方式の3パターン)

**主な設計上の決定**:

1. **アカウント既存/未取得の判定はシステム側では行わない**
   - 被招待者の招待承諾画面で「お持ちの方はログイン / 未取得の方は
     新規作成」を選択させる方式とした
   - 招待メール発行時のメールアドレス照合や、中間判定画面が不要

2. **団体招待での1会員1団体制約は二段階チェック**
   - 招待時のバリデーションと承諾時のバリデーションの両方で実施
   - 招待発行後に被招待者が他団体に加入する可能性に備える

3. **コース招待には招待URL方式を含む**
   - 自団体会員から選択 / メール招待 / URL発行 の3方式
     (主催者本人・団体管理者ともに使用可)
   - URL方式は被招待者が事前に確定しないため、A-CM1 一覧で
     「URL発行中」として1レコードのみ表示する特別な扱い
   - URLを受け取った人が**主催者と同じ団体所属の会員ならば承諾可、
     それ以外はエラー**

4. **コースメンバーは主催者と同じ団体所属の会員のみ**
   - コース主催者は団体所属を前提とする(無所属会員はコースを作らない)
   - コースメンバー(共同編集者)も主催者と同じ団体に所属する必要がある
   - 他団体所属者・無所属会員は招待不可
   - 団体管理者の介入編集権限としても、この制約は外れない
   - 動機: コースは主催者の所属団体内での共同作業の場として閉じる

5. **招待状態の管理**
   - 状態 enum: 送信済 / URL発行済(コース招待のみ) / 承諾済 /
     辞退 / 期限切れ
   - 期限切れからは管理者の「再招待」操作で送信済に戻せる

詳細(具体例、エッジケース、UI挙動、設計判断が必要な残論点)は
上記2ドキュメントを参照。
