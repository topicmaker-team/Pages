# 第1部: 9つの論点の決着サマリ

## 論点1: 公開設定の伝播方式の違い(=設計思想の整理)

**問題**: コースの公開設定変更時はユニットの公開設定の DB 値を上書きするが、団体の外部公開ロックを変更しても DB 値は変更せず動的ロジックで処理する。この差は問題ないか?

**決着**: **現状の仕様のまま**(混在ではなく、本質的な区別を反映している)

- 名称が示す本質的区別:
  - **「ロック」** = 規則(ルール) → 動的に評価される、属性は書き換えない
  - **「公開ステータス」** = 属性 → 実体的に値を持ち、実体的に書き換わる

- コース帰属切替時の「公開値の上書き」は**属性更新**として正当(投稿者の意思を反映する変更操作)
- 団体ロックの「実効的に格下げ」は**規則による評価結果**(閲覧時の振る舞い)であって属性変更ではない
- 規則違反は属性変更時にも閲覧時にも検出される(両方の場面で評価)

**仕様反映**: (α)設計思想を明示するセクションを追加(本文には書き込み済み or 追加予定)

## 論点2-1: 団体が削除された場合

**決着**: **(B-1)** 会員無所属化、団体所有コース・帰属ユニット削除、コース非帰属の個人ユニットは会員に残る

- 会員 → 無所属化(アカウント残る)
- 団体所有コース(団体所属メンバーが主催するコース) → 削除
- 削除されるコースに帰属していたユニット → 削除
- コース非帰属の個人ユニット → 会員に残る
- メディアファイル(削除対象コース・ユニットに関連) → S3 から削除
- ユーザーは事前に UC-23 等で退避可能(コンテンツの退避はユーザー責任)
- 削除前の確認モーダルで警告

## 論点2-2: コースが削除された場合

**決着**: **(a)** コース帰属ユニットも連鎖削除

- コース削除 → コース帰属ユニットも連鎖削除
- 配下ユニットの音声・画像メディア → S3 から削除
- コースメンバー関係 → 削除(関係レコードのみ)
- お気に入り・コメント(コース対象、コース帰属ユニット対象、その配下のパート対象、デッキ対象) → 削除
- 強い警告モーダル必須(配下のN件のユニットも削除されます、メンバーが投稿したM件のユニットも含まれます等)
- 削除前にユーザー(主催者・メンバー)が UC-23 等で退避する責任

## 論点2-3: 団体管理者(会員)が削除/離脱される場合

**決着**: **(I)** 他の管理者がいる場合のみ削除/離脱/ロール変更可

- **削除/離脱/ロール変更**いずれも、最後の管理者(自分が抜けると管理者0人になる)はエラー
- 最後の管理者は他の会員を管理者に昇格させてから操作する必要
- 一般会員が0人の団体で唯一の管理者が抜けたい場合は、団体削除を選ぶ(論点2-1の B-1 を経由)

## 論点2-4: コース主催者(会員)が離脱される場合

**決着**: **(II)** 離脱前に主催者を他会員に移譲(本フェーズで実装)

- 主催者の団体離脱前に、主催コースの主催者を別の人(同団体内の他会員)に移譲しなければならない
- 移譲先は同じ団体の所属会員のみ(主催者制約と同じ)
- 移譲が完了するまで離脱操作はエラー
- **新規UC-15a(仮称): コースの主催者を他会員に移譲する** を本フェーズで実装

## 論点3+4: UC-05「会員を削除する」の意味と enhack アカウントとの関係

**決着**: 

- UC-05「会員を削除する」 = **団体からの追い出し**(団体離脱の強制実行)
  - 主アクター: 団体管理者
  - 効果: Account.所属団体 を null にする(会員は無所属化)
  - 会員アカウントは残る、enhack ログイン継続可能
  - 制約:
    - 対象会員が団体管理者の場合は論点2-3 (I) に従う
    - 対象会員が主催コースを持つ場合: 主催者移譲が必要
- **新規UC-07: 本人による団体脱退**を追加
  - 自発的脱退時は **本人**が事前に主催者移譲(コースを持つ場合)
  - 追い出し時は **管理者が代理**で主催者移譲(管理者の介入権限の延長)
- enhack アカウント自体の削除は本人のみ実行可能、本仕様書のスコープ外

## 論点5: パート/デッキ/カードのID付与とエクセル更新

**決着**: 公開ID と 編集ID を分離。エクセルに編集ID列追加(先頭行記入ルール)

- **公開ID**(URL用、外部公開): パート、デッキに付与。カード、センテンスは持たない。
- **編集ID**(表形式ファイルの行同定用、内部のみ): パート、デッキ、カード、センテンス すべての階層に付与
- エクセル列構成: 10列 → **14列**(編集ID 4列を先頭に追加)

```
パート編集ID / デッキ編集ID / カード編集ID / センテンス編集ID /
チャプター名 / シーン名 / 話者 / 英会話 / 語順訳 / 和訳 / チェック /
音声ファイル / デッキ画像 / カード画像
```

- 記入ルール:
  - パート/デッキ/カードの編集ID: その階層の**先頭行のみ**記入(他行は空欄)
  - センテンス編集ID: **各行に記入**(センテンスはエクセルの行と1対1対応)
- 初回投稿時はすべて空欄、システムが投稿成功時に編集ID(と公開ID)を発行
- 更新時は、DLされた編集ID入りのエクセルを編集してアップロード
- ガイド: 投稿者には「編集ID列は触らない」ことを伝える

## 論点6: メディアパックの単位ダウンロード

**決着**: メディアパック単位で一括差し替え。S3 に zip 原本を別ディレクトリ(packs/)に保存

- メディア管理は **zip ファイル単位の一括差し替え**(個別ファイル単位の差分処理は行わない)
- ユーザーフロー:
  1. 現在の zip をダウンロードして手元で編集
  2. 編集済みの zip を再アップロードして全置換
- S3 構造:

```
s3://{bucket}/
  {ユーザーID}/
    {ユニットID}/
      {ファイル名}                  ← 展開後の個別ファイル(配信・参照用)
      packs/
        audio.zip                   ← アップロードされた最新の音声zip
        images.zip                  ← アップロードされた最新の画像zip
```

- 概念モデルの `音声パック.download()` `画像パック.download()` メソッドは、**packs/ 配下の zip 原本を返す機能**として正当化される
- アップロード時: zip を `packs/` に保存 + 中身を展開して個別ファイルにも保存
- 再アップロード時: 旧 zip と旧個別ファイル群をすべて削除 → 新 zip と新個別ファイル群を保存
- これにより zip と個別ファイル群は常に整合性が保たれる

## 論点7: S3 アップロード元

**決着**: **(A)** サーバー経由(プロキシ方式)

- クライアント → アプリサーバー → S3
- バリデーションを確実にサーバー側で実行
- AWS 認証情報がクライアントに漏れない
- 実装シンプル
- 将来パフォーマンス問題が顕在化したら署名付きURL方式への移行を検討

## 論点8: S3 データの公開範囲

**決着**: フェーズ別

- **フェーズ1(トピックメーカー社内ドッグフーディング)**: S3 バケットは**全公開**で OK
- **フェーズ2(他団体受け入れ可能化)以降**: 署名付きURL方式の導入
  - メディアファイルの URL 形式が変わることを承知
  - クライアント側でメディアURLをハードコードしない設計を推奨
  - 外部公開ロック機能の真の効果はフェーズ2の署名付きURL導入で発揮される

## 論点9: プレビュー時のメディア・自動生成タイミング

**決着**: プレビュー廃止、「**非公開で本登録 → A-MU2 でレビュー → 公開**」フローに変更

- A-MU3 の「投稿内容をプレビュー」ボタンは**削除**
- A-MU2 の「プレビュー(学習者視点)」は**残す**(これは登録された完成ユニットへの学習者視点アクセス)
- 処理フロー:

```
1. エクセル + zip アップロード
2-6. パース・バリデーション
7. 投稿者が「投稿する」(プレビュー画面なし)
8. ユニットID発行、DB保存(初期状態は「非公開」固定)
9. S3転送、TTS/AI画像生成ジョブ投入(非同期)
10. A-MU2 編集画面に遷移
11. 自動生成完了を待つ(進捗表示)
12. A-MU2 で投稿者がレビュー(再生・表示確認、必要なら修正)
13. レビュー完了後、公開状態を「非公開」から目的の値に変更
14. これで実質的に「公開」状態になる
```

- 自動生成中の状態を A-MU2 上で表示する仕組み(進捗バー、または「生成中」バッジ)
- 自動生成完了通知(または、投稿者が自分でリロードして確認)
- 「非公開」が事実上の下書き状態として機能する(enum としては既存の値そのまま)

# 補足: 直前で議論された事項のメモ

これらは決着済みだが、改訂作業時に細かく注意すべき点として:

1. **公開IDと編集IDの実装上の表記**: 概念モデル図では `+String *公開ID` `+String *編集ID` のように表記する(`*` は他クラスから参照される識別子の慣例)
2. **論点1の(α)設計思想の文章**: 仮文案として用意済み(必要に応じて調整)
3. **メディアパックの zip と個別ファイルの整合性**: 一括差し替えなので常に同期、エッジケースなし
4. **論点3+4の運用上の補足**: enhack アカウント削除は本仕様書スコープ外、本書では「他者のアカウントを勝手に削除しない」と明示
5. **論点1で1点保留事項**: 規則変更時の既存違反コンテンツの扱い(α/β/γ/δ)は実装時のUX設計議論として保留中
