# ちょい英語ミニブック投稿システム — 投稿データ / メディアパック仕様

本書は、ユニット投稿(UC-20)および投稿データ更新(UC-22)で扱う、
投稿データ(エクセル) / 音声パック / 画像パック の構造と処理仕様を定める。

## 0. 関連仕様の前提

本書を読む前に、ユースケース仕様書(`ちょい英語ミニブック投稿システム_
ユースケース_画面一覧.md`)の以下のセクションを既知とする:

- 「教材階層のIDの二系統(公開ID と 編集ID)」
- 「S3 データの公開範囲(フェーズ別方針)」
- 「3. ユニット投稿」セクション冒頭の投稿フロー基本方針
  (「**本登録 → 自動生成 → A-MU2 でレビュー(必要なら修正)**」、登録前
  プレビュー画面は提供しない)

## 1. エクセル構造

### 1.1 シート構成

- シート数: 1(シート名は任意、実装上は先頭シートのみを読む)
- 1行目: ヘッダー行
- 2行目以降: データ行(1行=1センテンス)

### 1.2 列定義(14列)

エクセルは **編集ID 4列** + **表示用 10列** = **計14列** で構成する。
編集ID列は内部処理用で、投稿者は触らない(後述 §1.4)。

| # | 列名 | 対応する概念モデル属性 | 記入粒度 | 必須 | 区分 |
|---|---|---|---|---|---|
| 1 | パート編集ID | パート.編集ID | パートの**先頭行のみ**記入 | システム | 編集ID |
| 2 | デッキ編集ID | デッキ.編集ID | デッキの**先頭行のみ**記入 | システム | 編集ID |
| 3 | カード編集ID | カード.編集ID | カードの**先頭行のみ**記入 | システム | 編集ID |
| 4 | センテンス編集ID | センテンス.編集ID | **各行に記入**(センテンスは行と1対1) | システム | 編集ID |
| 5 | チャプター名 | デッキ(識別キー) | 各デッキの全行に同じ値 | 必須 | 表示用 |
| 6 | シーン名 | カード(識別キー) | 各カードの全行に同じ値 | 必須 | 表示用 |
| 7 | 話者 | センテンス(表示用情報) | 各行固有 | 任意 | 表示用 |
| 8 | 英会話 | センテンス.英文 | 各行固有 | 必須 | 表示用 |
| 9 | 手順3で作成した語順訳 | センテンス.直訳 | 各行固有 | 任意 | 表示用 |
| 10 | 和訳 | センテンス.和訳 | 各行固有 | 任意 | 表示用 |
| 11 | チェック | 投稿者側の作業フラグ(DBには保存しない) | 各行固有 | 任意 | 表示用 |
| 12 | 音声ファイル | センテンス.音声ファイルID に相当 | 各行に記入(1発話=1音声) | 条件付き | 表示用 |
| 13 | デッキ画像 | デッキ.画像ファイルID に相当 | **デッキの先頭行のみ**記入、他行は空欄 | 条件付き | 表示用 |
| 14 | カード画像 | カード.画像ファイルID に相当 | **カードの先頭行のみ**記入、他行は空欄 | 条件付き | 表示用 |

列12〜14 の「条件付き」は、TTS/AI画像の自動生成が ON の場合は空欄可、
OFF の場合は必須となる(詳細は §3)。

### 1.3 階層の識別ルール

- **デッキの識別**: 「チャプター名」の値が変わる境界で新しいデッキが始まる
- **カードの識別**: 「シーン名」の値が変わる境界で新しいカードが始まる
- **センテンスの識別**: データ行1行 = 1センテンス
- **デッキ・カードの表示順**: エクセル内での登場順をそのまま表示順とする

### 1.4 編集IDの記入ルール

編集IDは、投稿データ更新時の create / update 判定および
階層構造の同定に用いる内部識別子である(詳細はユースケース仕様書
「教材階層のIDの二系統」参照)。

**記入ルール**:

- **パート編集ID / デッキ編集ID / カード編集ID**: その階層の
  **先頭行のみ**記入(他行は空欄)
- **センテンス編集ID**: **各行に記入**(センテンスはエクセルの行と
  1対1対応)

**初回投稿時(UC-20)**:

- 4列すべてを空欄でアップロード
- システムが投稿成功時に編集IDを発行し、内部的に各レコードに付与

**更新時(UC-22)**:

- UC-23 で DLされたエクセルには、4列すべてに編集IDが反映されている
- 投稿者はこのエクセルを編集してアップロードする
- システム側の判定:
  - 編集ID列が空欄の行 → **新規作成**(create)
  - 編集ID列に値がある行 → **更新**(update)
- 投稿者向けガイド: **「編集ID列は触らないでください」** とエクセル
  ヘッダのコメントおよびUI上のヘルプテキストで案内する

**先頭行ルール違反のバリデーション**:

- パート/デッキ/カードの編集ID列で、先頭行以外に値が入っていれば
  バリデーションエラー
- センテンス編集ID列は、空欄(create)または値あり(update)の
  どちらでも可

### 1.5 デッキ画像・カード画像の先頭行ルール

```
パート編集ID | デッキ編集ID | カード編集ID | センテンス編集ID | チャプター名 | シーン名 | ... | デッキ画像 | カード画像
p_001        | d_001        | c_001        | s_0001           | チャプター1  | シーン1  | ... | c1.png    | c1s1.png     ← デッキ先頭 かつ カード先頭
             |              |              | s_0002           | チャプター1  | シーン1  | ... |           |              ← 2行目以降は空欄
             |              |              | s_0003           | チャプター1  | シーン1  | ... |           |              ← 同上
             |              |              | s_0004           | チャプター1  | シーン1  | ... |           |              ← 同上
             |              | c_002        | s_0005           | チャプター1  | シーン2  | ... |           | c1s2.png     ← カード先頭(デッキは継続)
             |              |              | s_0006           | チャプター1  | シーン2  | ... |           |              
...
             | d_002        | c_009        | s_0033           | チャプター2  | シーン1  | ... | c2.png    | c2s1.png     ← デッキが切り替わる
```

- 先頭行以外の行に画像ファイル名が書かれていた場合は**バリデーションエラー**
- 先頭行が空欄だった場合は「このデッキ/カードに画像なし」と解釈

### 1.6 ファイル名の命名規則

投稿者の自由(拡張子と S3 に許容される文字集合のみ強制)。

- 許容文字: 英数字、アンダースコア(`_`)、ハイフン(`-`)、ドット(`.`)
- 禁止: 空白、日本語、スラッシュ、その他特殊文字
- 音声: `.mp3` / `.wav` / `.m4a` 等(実装で定義)
- 画像: `.png` / `.jpg` / `.jpeg` / `.webp` 等
- 動画(将来): `.mp4` 等

同じファイル名を複数行(複数センテンス/複数カード)が参照しても良い。
その場合、S3上には1つのオブジェクトのみ存在し、DB上の複数レコードが
同じキーを参照する。

## 2. メディアパック(zip構造)

本章は **メディアパックモード** を選択した場合の zip ファイル構造を
定める。自動生成モードを選択した場合は zip のアップロードは不要(§3
参照)。

メディアパックは **zip ファイル単位の一括差し替え** が基本方針である。
ユーザーが個別ファイル単位で差分処理を行う必要はない。

### 2.1 一括差し替えの原則

- 投稿時 / 更新時にアップロードされた zip ファイルが、ユニットの
  メディアの最新版となる
- 既存ファイルとの個別比較は行わない
  - 例: 64件中3件だけ差し替える場合でも、64件すべてを含む zip を再アップロードする
  - これにより整合性管理がシンプルになり、エラー時の復旧も容易
- 投稿者向けフロー:
  1. UC-23 で現在の zip をダウンロードして手元に保持
  2. 必要な編集を加えた新しい zip を作成
  3. 新しい zip を再アップロードして全置換

### 2.2 音声パック

```
audio.zip
├── c1_s1_u1.mp3
├── c1_s1_u2.mp3
├── c1_s1_u3.mp3
├── ...
└── c2_s8_u4.mp3
```

- フラット構造(サブディレクトリなし)を原則とする
- ファイル名はエクセルの「音声ファイル」列と完全一致
- zip内で重複するファイル名は不可

### 2.3 画像パック

```
images.zip
├── c1.png              ← デッキ画像
├── c2.png              ← デッキ画像
├── c1s1.png            ← カード画像
├── c1s2.png
├── ...
└── c2s8.png
```

- フラット構造を原則
- デッキ画像とカード画像は同一zip内に混在して良い
- ファイル名はエクセルの「デッキ画像」「カード画像」列と完全一致

### 2.4 動画の扱い(将来拡張)

画像パックに動画ファイルを含める形を想定するが、本フェーズでは対象外。
`画像パック.メディアタイプ : 静止画 / 動画` enum との対応は、将来
「画像パック」を「メディアパック」に概念拡張するタイミングで整備する。

## 3. メディア供給モード(自動生成 / メディアパック)の排他選択

ユニットの音声・画像メディアは、それぞれ独立に **自動生成モード** と
**メディアパックモード** のいずれかで供給する。両モードは **排他選択** で
あり、行ごとの混在は許可しない(ファイル名衝突や運用上の複雑さを
避けるため)。

### 3.1 モードの選択

A-MU3 新規投稿画面 / A-MU2 編集画面に以下のラジオを配置する:

- **音声メディア**:
  - メディアパック(zip)をアップロード
  - TTS で自動生成
- **画像メディア**:
  - メディアパック(zip)をアップロード
  - AI画像で自動生成

音声と画像は独立に選べる(音声=TTS / 画像=パック の組み合わせも可、
逆も可)。

### 3.2 メディアパックモードの挙動

- 投稿者はエクセルの該当列(音声ファイル / デッキ画像 / カード画像)を
  **すべて記入**し、対応する zip をアップロードする
- エクセルと zip の参照整合性は厳格にチェック(§4.2)
- ファイル名衝突や余剰ファイルはエラーで投稿拒否

### 3.3 自動生成モードの挙動

- 投稿者はエクセルの該当列を **空欄のままにする**(自動生成対象)
- システムが生成し、各ファイルにシステム生成名を付与する
  - 命名規則例: `_gen_sent_00001.mp3`, `_gen_deck_001.png`,
    `_gen_card_001.png` のような予約プレフィクス
  - 命名規則の正確な仕様は実装フェーズで詳細設計する(当面は
    予約プレフィクス方式で衝突回避を保証)
- 生成完了後、システムがエクセル上の該当列にもシステム生成名を
  反映する(UC-23 のダウンロード時に整合した状態で取得できる)

### 3.4 モードの混在(行レベルの混在)は不可

「同じユニット内で、音声のうち一部の行は zip 由来・一部は TTS 生成」
のような行レベル混在は **許可しない**。混在を許すとファイル名衝突の
リスクが生じ、UC-23 のダウンロードや UC-22 の更新時の処理が複雑化
するため。

ただし、**音声と画像は独立に選べる**ので、「音声=zip / 画像=AI画像」
のような組み合わせは可能。

### 3.5 モード切替(UC-22)

A-MU2 上でメディア供給モードを後から切り替えることができる。
切替時の挙動は以下:

| 切替前 | 切替後 | 挙動 |
|---|---|---|
| 自動生成 | メディアパック | 既存の生成物(展開ファイル)を **全削除** → アップロードされた新 zip を `packs/` に保存し展開ファイルを配置 |
| メディアパック | 自動生成 | 既存の zip 原本と展開ファイルを **全削除** → 自動生成ジョブを投入 |
| 自動生成 | 自動生成(再生成) | 既存の生成物を **全削除** → 自動生成ジョブを再投入 |
| メディアパック | メディアパック(差し替え) | 既存の zip 原本と展開ファイルを **全削除** → 新 zip を保存し展開ファイルを配置 |

いずれのケースも「**既存をすべて削除して入れ替える**」というシンプルな
原則で統一する。エクセルの該当列(音声ファイル / デッキ画像 / カード画像)
の値は、新モードに応じてシステムが書き換える(自動生成モードに切り替えた
場合は空欄に、生成完了後にシステム生成名に更新)。

**運用例**: 自動生成で本登録 → UC-23 で生成物をダウンロード → 手元で
気に入らないファイルを別途調達したファイルで差し替え → エクセルの
該当列も手元で編集 → A-MU2 でメディアパックモードに切り替え + 編集
済み zip と編集済みエクセルをアップロード → メディアパックモードで
更新完了。

### 3.6 自動生成完了のタイミング

自動生成は **A-MU3 での本登録(「投稿する」または「下書き保存」)の後**、
非同期ジョブで実行される(§6 参照)。投稿者はジョブ完了を A-MU2 編集
画面で待つ(進捗表示)。

サーバー側生成処理を伴うため、登録前プレビューは原理的に成立せず、
A-MU3 では登録前のプレビュー画面を提供しない。生成された音声・画像の
確認は A-MU2 で行う。

詳細は §6 / §7 の処理フローおよびユースケース仕様書「3. ユニット投稿」
セクション冒頭の基本方針を参照。

## 4. バリデーション

A-MU3 投稿時 / A-MU2 差分適用時に実施する検証。

### 4.1 構造チェック(エクセル単体)

- 必須列(チャプター名、シーン名、英会話)が全て埋まっているか
- 編集ID列の記入ルール(§1.4)に違反していないか
  - パート/デッキ/カード編集ID列が、先頭行以外の位置に書かれていないか
  - センテンス編集ID列以外の編集ID列が、先頭行で空欄でない場合に
    階層内のID整合性が取れているか(同一階層内で同一ID)
- デッキ画像・カード画像が先頭行以外に書かれていないか(§1.5)
- ファイル名に禁止文字が含まれていないか(§1.6)

### 4.2 整合性チェック(エクセル × zip)

メディアパックモードのときに、**不整合はエラーとして投稿を拒否する**
(自動補完は行わない)。

不整合と判定されるケース:

1. エクセルで参照されているファイル名が zip にない
2. zip にあるがエクセルから参照されていない余剰ファイル
3. 同じファイル名が zip 内で重複
4. ファイル拡張子が許容リスト外

自動生成モードのときは、エクセルの該当列はすべて空欄でなければならない
(モード一貫性チェック、§4.5)。

エラー時の UI 表示例(既存ワイヤーフレームの警告バー):

> エラー:
> - 音声ファイル `c1_s1_u1.mp3` がエクセルで参照されていますが、zipに見つかりません
> - 画像ファイル `extra.png` がzipに含まれていますが、エクセルから参照されていません
> - デッキ「チャプター2」のカード画像がシーン先頭行ではない位置(行15)に記載されています

### 4.3 スキーマ制約チェック(UC-14 連携)

- 最大デッキ数、最大カード数、デッキ内最大カード数 を超過していないか
- 超過時はエラーで投稿拒否

### 4.4 編集ID の存在性チェック(UC-22 更新時)

- エクセルに記入されている編集IDが、当該ユニット配下に実在するか
- 他ユニットの編集IDが混入していないか
- 不正な編集ID参照はエラーとして拒否

### 4.5 モード一貫性チェック

メディア供給モード(自動生成 / メディアパック)とエクセル・zip の
内容が整合しているかを検証する。

| モード | 期待される状態 | 違反時の挙動 |
|---|---|---|
| 音声 = メディアパック | エクセルの「音声ファイル」列が全行記入されている / 音声zipがアップロードされている | エラー: 「音声メディアパックモードでは音声ファイル列を全行記入し、音声zipをアップロードしてください」 |
| 音声 = 自動生成 | エクセルの「音声ファイル」列が全行空欄 / 音声zipなし | エラー: 「音声自動生成モードでは音声ファイル列は空欄にし、音声zipはアップロードしないでください」 |
| 画像 = メディアパック | エクセルの「デッキ画像」「カード画像」列がデッキ・カード先頭行に記入されている / 画像zipがアップロードされている | エラー: 同上(画像版) |
| 画像 = 自動生成 | エクセルの「デッキ画像」「カード画像」列が全行空欄 / 画像zipなし | エラー: 同上(画像版) |

排他選択を強制することで、混在状態を防止する。

## 5. S3 格納規則

### 5.1 アップロード方式

クライアントから直接 S3 にアップロードするのではなく、
**アプリサーバー経由(プロキシ方式)** で S3 に転送する。

```
クライアント → アプリサーバー → S3
```

理由:

- バリデーション(§4)をサーバー側で確実に実行できる
- AWS 認証情報がクライアントに漏れない
- 実装がシンプル(単一の入口でログ/監視を集約)

将来パフォーマンス問題が顕在化した場合は、署名付きURLによる
クライアント直接アップロード方式への移行を検討する。

### 5.2 公開範囲(フェーズ別方針)

S3 の公開範囲はフェーズに応じて段階的に厳格化する。

- **フェーズ1(トピックメーカー社内ドッグフーディング)**:
  S3 バケットは**全公開**。メディアURLは固定の S3 公開URL で
  クライアントから直接参照可能
- **フェーズ2(他団体受け入れ可能化以降)**: **署名付きURL方式**
  に移行。メディア配信時に動的に署名付きURLを発行し、有効期限と
  閲覧権限のチェックを伴う形で配信する

詳細はユースケース仕様書「S3 データの公開範囲(フェーズ別方針)」を参照。
クライアント側でメディアURLをハードコードしない設計を推奨。

### 5.3 キー構造

```
s3://{bucket}/
  {ユーザーID}/
    {ユニットID}/
      {ファイル名}                   ← 展開後の個別ファイル(配信・参照用)
      packs/
        audio.zip                    ← メディアパックモードの場合の原本zip
        images.zip                   ← メディアパックモードの場合の原本zip
      _cover.{ext}                   ← ユニット表紙画像(予約名)
```

- `ユーザーID`: Account の安定識別子(ULID等)
- `ユニットID`: Unit の安定識別子
- ファイル名は、メディアパックモードではエクセルで指定されたものを
  そのまま使う。自動生成モードではシステム生成名(`_gen_...` 等の
  予約プレフィクス、§3.3 参照)

### 5.4 モード別の格納方針

#### 5.4.1 メディアパックモードの場合

- アップロードされた **zip 原本** を `packs/audio.zip` または
  `packs/images.zip` として保存
- zip を **展開した個別ファイル** を `{ユニットID}/` 直下に配置
  (配信・参照用)
- 両者は常に同一内容で同期する
- UC-23 のダウンロードは `packs/` 配下の zip 原本を返す
- 概念モデルの `音声パック.download()` `画像パック.download()`
  メソッドは、この zip 原本を返す機能として実装される

#### 5.4.2 自動生成モードの場合

- システムが生成した個別ファイルを `{ユニットID}/` 直下に配置
  (配信・参照用)。`packs/` 配下に zip 原本は作成しない
- UC-23 のダウンロードでは、ダウンロード時点で **動的に zip を生成**
  して返す。生成内容:
  - 現時点でエクセルから参照されているすべての音声/画像ファイル
  - ファイル名はシステム生成名そのまま(命名規則は §3.3 参照)
- この動的生成された zip は S3 に保存しない(都度生成)。または、
  キャッシュ目的で一時的に保存する選択肢もあるが、実装フェーズで詳細
  設計する

#### 5.4.3 モード切替時の整合性維持

§3.5 の表の通り、モード切替時はすべて **「既存をすべて削除して入れ替え」**
で処理する:

- 旧モードのファイル(`packs/` 配下の zip と展開ファイル、または
  自動生成の個別ファイル)をすべて削除
- 新モードに応じて新しいファイルを配置(zip保存+展開、または自動生成
  ジョブ投入)

これにより、複数モードの中間状態が S3 上に残ることはない。

### 5.5 ユニットID の発行タイミング

- **A-MU3「投稿する」または「下書き保存」ボタン押下時**にユニットIDが
  確定する
- それまではS3への保存は発生しない
- A-MU3 では**エクセル + (必要なら)zip の一括アップロード**が必須
- 「投稿する」と「下書き保存」の違いは公開設定のみ:
  - 「投稿する」: 投稿者が選択した公開設定で本登録
  - 「下書き保存」: 公開設定を「非公開」固定で本登録
- いずれの場合も登録後ただちに自動生成ジョブが起動し、A-MU2 に遷移する
  (詳細は §6 を参照)

### 5.6 ユニット表紙画像

A-MU3/A-MU2 の UI 上で**エクセルとは別に**直接アップロードする
(現行ワイヤーフレームの「カバー画像」フィールド通り)。S3キーは:

```
s3://{bucket}/{ユーザーID}/{ユニットID}/_cover.{ext}
```

アンダースコア始まりの予約ファイル名を使い、投稿者が同名のファイルを
エクセルから参照していた場合は警告(ただしこの予約ファイル名は
投稿者には公開しないので、通常衝突しない)。

### 5.7 UC-22 差分更新時の挙動

- ユニットIDは既に確定しているので、S3キーは維持される
- **エクセルの差分処理**: 編集IDによって行単位で create / update を
  判定し、DB を更新する(個別行レベルでの差分処理)
- **メディアの処理**: モード別に以下のように扱う(§3.5 のモード切替
  ルールも参照)

  | 更新内容 | 挙動 |
  |---|---|
  | モード変更なし、zip 再アップロードなし(メディアパック) | メディアは現状維持 |
  | モード変更なし、zip 再アップロード(メディアパック → メディアパック) | 旧 zip と展開ファイルを全削除 → 新 zip 保存・展開 |
  | モード変更なし(自動生成 → 自動生成、パラメータ変更等) | 既存生成物を全削除 → 自動生成ジョブを再投入 |
  | モード変更あり(自動生成 → メディアパック) | 既存生成物を全削除 → 新 zip 保存・展開 |
  | モード変更あり(メディアパック → 自動生成) | 旧 zip と展開ファイルを全削除 → 自動生成ジョブを投入 |

- 差分プレビューでこれらの変化を投稿者に表示する(モード変更時は
  特に強く警告)

## 6. 処理フロー(A-MU3 新規投稿)

投稿フローは「**本登録 → 自動生成 → A-MU2 でレビュー(必要なら修正)**」を
基本方針とする。サーバー側生成処理を伴うため、登録前のプレビュー画面は
設けない(生成物がまだ存在しないため、原理的に成立しない)。

メディア供給モードは音声・画像それぞれ独立に選択し、両モードは排他
(§3 参照)。以下のフローでは「メディアパックモード」と「自動生成モード」
を分岐させて記述する。

```
1. 投稿者がエクセルをアップロード
   ↓
2. 投稿者が音声・画像それぞれのメディア供給モードを選択
   - メディアパックモード: 対応するzipを同時にアップロード
   - 自動生成モード: zipはアップロードしない
   ↓
3. システムがエクセルをパース
   ↓
4. 構造チェック(§4.1)
   ↓ NG → エラー表示、中断
   ↓ OK
5. モード一貫性チェック(§4.5)
   ↓ NG → エラー表示、中断
   ↓ OK
6. zip がある場合は展開し、整合性チェック(§4.2)
   ↓ NG → エラー表示、中断
   ↓ OK
7. スキーマ制約チェック(§4.3)
   ↓ NG → エラー表示、中断
   ↓ OK
8. 投稿者が「投稿する」または「下書き保存」をクリック
   ↓
9. ユニットIDを発行
   ↓
10. エクセル内容をDBに永続化(編集IDを各レコードに付与)
    - ユニット → パート(1個固定) → デッキ → カード → センテンス
    - 公開状態:
      - 「投稿する」: 投稿者が A-MU3 で選択した公開設定で保存
      - 「下書き保存」: 「非公開」固定で保存
    - メディア供給モード(音声・画像それぞれ)もユニット属性として保存
    ↓
11. メディア配置(モード別):
    - メディアパックモード: zip 原本を packs/ に保存 → 展開して個別
      ファイルを配置
    - 自動生成モード: 自動生成ジョブを非同期に投入
      (個別ファイルは生成完了時に配置される)
    ↓
12. A-MU2 に遷移(ユニット詳細/編集画面)
    ↓
13. 投稿者が A-MU2 上で:
    - 自動生成完了を待つ(進捗表示。該当モードの場合のみ)
    - レビュー(再生・表示確認、必要なら修正)
    - 必要に応じて公開設定や料金などを編集して保存
```

ステップ9〜12までが「本登録」のスコープ。投稿者から見ると、A-MU3 で
ボタンを押した後はそのまま A-MU2 に遷移してコンテンツが揃うのを待つ
流れになる。

## 7. 処理フロー(A-MU2 差分更新)

```
1. A-MU2 画面で「⭳ 投稿データをダウンロード」クリック
   ↓
2. システムが現在のDB内容をエクセル形式で出力
   - 編集ID列に各レコードの編集IDを反映
   - 音声ファイル列、デッキ画像列、カード画像列は現在のS3キーを反映
     (自動生成モードの場合はシステム生成名が反映される)
   ↓
3. (必要なら)A-MU2 上で「メディアパックをダウンロード」クリック
   - メディアパックモード: S3 packs/ 配下の zip 原本を取得
   - 自動生成モード: 動的に zip を生成して取得(§5.4.2)
   ↓
4. 投稿者がエクセル(+必要なら zip)を手元で編集
   - 編集ID列は触らない
   - メディア供給モードを変更したい場合は、A-MU2 上で次のステップで
     モードを切り替える
   ↓
5. A-MU2 でメディア供給モードを選択し、編集後エクセル(+必要なら新しい
   zip)をアップロード
   - 自動生成モード → メディアパックモード への切替時は、ダウンロード
     した zip を手元で編集し直して再アップロードするのが一般的な運用
   ↓
6. システムがエクセルの差分を解析
   - 編集ID列が空欄の行 → create
   - 編集ID列に値がある行 → update
   - DLされた行のうちアップロードに含まれない行 → delete(削除はWeb UI経由のみ、§後述)
   ↓
7. バリデーション(§4、モード一貫性チェック §4.5 を含む)
   ↓ NG → エラー、中断
   ↓ OK
8. 差分プレビュー表示
   - エクセル変更サマリ(例: create 3 / update 12 / ignore 50)
   - メディア操作サマリ(モード切替の有無、zip再アップロードの有無、
     自動生成ジョブの再投入予定など)。モード変更時は強く警告
   ↓
9. 投稿者が「差分を適用」クリック
   ↓
10. DB 更新(モード変更があればユニット属性も更新)
    ↓
11. メディア処理(§5.7 の表に従う):
    - モード変更なし、メディアパック zip 再アップロード: 旧 zip と展開
      ファイルを削除、新 zip を保存・展開
    - モード変更なし、自動生成(再生成): 既存生成物を削除、自動生成
      ジョブを再投入
    - モード変更あり(自動生成 → メディアパック): 既存生成物を削除、
      新 zip を保存・展開
    - モード変更あり(メディアパック → 自動生成): 旧 zip と展開ファイル
      を削除、自動生成ジョブを投入
    - メディア処理なし(エクセルのみ更新): 何もしない
    ↓
12. 完了
```

> 注: 行単位の **削除** は本フローでは扱わない(エクセル経由ではなく
> A-MU2 の Web UI 上の操作で行う)。詳細はワイヤーフレーム参照。

## 9. 残論点 / 将来拡張

1. **ユニット表紙画像のS3キー衝突対策**: `_cover.{ext}` の予約名について、投稿者への公開範囲と衝突検知の実装詳細
2. **音声/画像の上限サイズ**: 1ファイルあたりの最大サイズ、zipの最大サイズ
3. **S3オブジェクトのライフサイクル**(フェーズ2): 非公開ユニットのメディアへの署名付きURLの発行、期限管理
4. **自動生成の非同期処理**: 投稿時に自動生成が完了しない場合、ユーザにどう見せるか(「生成中」バッジ等)。A-MU2 上での進捗表示の仕組み
5. **動画パック**: 将来的な拡張として概念モデルの `画像パック` を `メディアパック` に拡張
6. **ファイル名の大文字小文字**: S3キーは大文字小文字を区別するが、エクセルでの入力揺れ(Windows感覚で小文字化しがち)への対応
7. **メディアパックのバージョン管理**: 投稿データ(エクセル)はバージョン管理しない方針(UC-22注記)。メディアパック(zip原本)も同様にバージョン管理せず、最新版のみ保持(`packs/audio.zip`, `packs/images.zip` の単一名で上書き)
